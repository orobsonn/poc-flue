## Stage 0 — escopo concreto

**Goal**: capturar baseline numérico do v0.2 atual antes de mexer em qualquer coisa, pra que cada stage v0.3.X possa ser comparado.

**Entregas mínimas**:

1. `src/lib/metrics.ts` (~30-40 linhas) com função `recordRunMetrics(env, runId, payload)` que:
   - Recebe `{ runId, severity, divergences_detected, classifications_succeeded, candidates, buckets_active, latency_ms_total, llm_calls_count }`
   - Persiste em `monitor-runs/<runId>/metrics.json` no R2 via `fawWrite`
   - Inclui timestamp UTC
2. Instrumentar `monitor.ts` pra coletar e chamar `recordRunMetrics` no fim do pipeline:
   - `latency_ms_total = Date.now() - runStartTs`
   - `llm_calls_count` — contador incrementado em cada `s.skill(...)` (envolver em helper local ou contar manualmente nos 3 lugares: detect, classify+suggest, summarize)
3. `scripts/baseline-runs.ts` — roda smoke 5x sequencial, agrega `metrics.json` de cada run, imprime tabela: `run | duration | llm_calls | divergences_detected | classifications_succeeded | severity`. Salva agregado em `docs/V0.3-BASELINE.md` (gerado, com data + hora).
4. **Custo USD via Gateway API**: criar `scripts/fetch-cost.ts --run-id X` (ou flag `--last-n 5`) que puxa logs do AI Gateway via API e soma `cost` por run. Reusa o curl funcional que validamos em v0.2 — variáveis em `../../.dev.vars`. Endpoint: `https://api.cloudflare.com/client/v4/accounts/{account}/ai-gateway/gateways/{gateway}/logs?per_page=N`. Token precisa ter permissão **AI Gateway: Read** (já confirmado funcionando no v0.2).

**Critério de aceitação Stage 0**:
- 5 runs sequenciais com mesmo workload (semear D1 do zero, mesmo seed se possível)
- `docs/V0.3-BASELINE.md` com tabela de métricas consolidadas
- p50 + p95 calculados pra cada métrica
- Custo por run visível e somado
- Commit único: `chore(metrics): instrumenta baseline v0.2 pra comparacao com stages v0.3`

## Gotchas conhecidos (do v0.2, não esquecer)

1. **`pocs/auditor/.wrangler/state` ≠ `pocs/auditor/dist/.wrangler/state`** — o dev server roda do `dist/`, então migrations e queries via `wrangler d1 execute` precisam ser feitas com `cwd=dist/`. Aplicar migration na raiz não propaga.
2. **Endpoint Flue exige `/agents/<name>/<id>`** — não use `/agents/<name>` sozinho (404). `seed-baseline.ts` e `smoke.ts` já estão corretos; replicar o padrão em scripts novos.
3. **Hot-reload de `wrangler.toml` é inconsistente** — se mudar `MODEL_MAIN`, talvez precise reiniciar o dev server pra propagar.
4. **`pi-ai` só tem alguns modelos `@cf/` registrados** — verificar em `node_modules/@mariozechner/pi-ai/dist/models.generated.js` antes de tentar. Lista atual: `gemma-4-26b-a4b-it`, `llama-4-scout-17b-16e-instruct`, `kimi-k2.5`, `kimi-k2.6`, `nemotron-3-120b-a12b`, `gpt-oss-20b`, `gpt-oss-120b`, `glm-4.7-flash`. Modelo atual em uso: `kimi-k2.6`.
5. **Token CF do `.dev.vars` precisa ter `AI Gateway: Read`** — confirmado funcionando no v0.2 quando o user adicionou a permissão. Se quebrar, `curl /user/tokens/verify` confirma se token está válido (e se tiver, problema é de scope, não de validade).
6. **Kimi K2.6 demora ~4-5min por run completo** (vs ~40s do gpt-oss-120b). Plan accordingly em scripts que rodam smoke múltiplas vezes — `baseline-runs.ts` com 5 execuções vai levar ~25min só de LLM. Evitar timeouts curtos.

## Decisões abertas pra você considerar antes do Stage 1

Ler estas perguntas antes de tocar código no Stage 1 — algumas afetam o design da instrumentação do Stage 0:

1. **Custo cap**: o doc V0.3-EVOLUTION.md sugere `max_cost_usd = 0.50` por run no Stage 1. Verificar se isso é realista (com Kimi cobrando ~$X/M tokens, 30 tool_calls cabem nesse cap?). Se Kimi é caro, considerar voltar pro `gpt-oss-120b` (Harmony bug pode não afetar tool-use puro, vale testar isolado) ou usar `kimi-k2.5` (mais barato).
2. **Tool whitelist no Stage 1**: V0.3 sugere `read` limitado a 3 prefixes (`expected-reasoning/`, `agents-config/`, `references/`). Implementar esse rate limit dentro do helper Flue ou via prompt instruction? (Sugestão: ambos — instrução clara + validação em código, defesa em profundidade.)
3. **Onde colocar a lógica de ladder**: agente Flue novo será um novo agent em `.flue/agents/auditor-agentic.ts` ou substitui o `monitor.ts` direto via flag de ambiente (`AUDIT_MODE=pipeline|agentic`)? Sugestão: arquivo separado, easier rollback, deixa pipeline funcionando como controle.
4. **Refatorar SKILL.md das 4 skills pra remover `result: Schema`** — quando o agente as invoca como tools, não pode usar marker-based extraction (ou pode? checar Flue docs). Pode ser que as skills precisem de versões "agentic" separadas.

Resposta dessas perguntas pode mudar o design do Stage 0 — particularmente decisão (1) sobre modelo afeta o custo medido e portanto o critério go/no-go de cada stage.
