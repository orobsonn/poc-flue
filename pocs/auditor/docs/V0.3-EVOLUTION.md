# auditor POC v0.3 — Validação do Flue como framework agêntico

## Tese a validar

**Flue permite construir agentes que raciocinam e agem em loop com poucas linhas de código, fornecendo valor real comparado a um pipeline imperativo equivalente.**

Se a tese é falsa, Flue é um wrapper sobre `fetch` com peso de framework — podemos remover sem perda. Se é verdadeira, devemos ver: (a) menos código pra mesma funcionalidade, (b) emergência de comportamentos que o pipeline não captura, (c) trade-offs aceitáveis em custo e latência.

## O que v0.2 NÃO validou

A v0.2 entregou pipeline ponta-a-ponta funcionando, mas:

- **Loop agêntico desativado** — todas as chamadas usam `result: Schema`, que reduz a skill a "prompt-and-extract" single-shot. Tools (`read`, `write`, `bash`, etc.) injetadas pelo Flue nunca foram exercitadas.
- **Skill discovery não usado** — `monitor.ts` invoca skills hardcoded em ordem fixa. O modelo nunca decidiu qual skill chamar.
- **References (`references/<x>.md`) não acessadas** — o padrão progressive-disclosure das Anthropic Skills nunca foi exercitado. Quando suggest-adjustment tentou (no run com gpt-oss-120b), bateu no Harmony channel bug.
- **Sessions stateless** — cada chamada constrói contexto do zero; histórico de mensagens não é exercitado.

Em outras palavras, v0.2 usa Flue como wrapper conveniente sobre Workers AI, não como framework agêntico. **A tese central do POC continua não testada.**

## Estrutura do experimento

Ladder progressivo de 4 stages. Cada stage move 1 fronteira de responsabilidade do `monitor.ts` (TypeScript determinístico) pro **agente Flue** (LLM com tools). Cada stage tem:

- **Hipótese específica** (o que esperamos provar)
- **Mudanças técnicas** (escopo concreto)
- **Métricas vs baseline v0.2** (custo, latência, qualidade, robustez)
- **Critério go/no-go** (se falhar, abandona o stage e o ladder)

Baseline v0.2 já está documentado: ~277s/run com Kimi K2.6, ~13 LLM calls, 3 divergências detectadas + classificadas + sugeridas em PR estruturado, severidade computada por regra SQL+LLM, custo ~$X (medir antes do v0.3).

### Stage 1 — Handoff post-bucketing (mínimo viável agêntico)

**Hipótese**: Flue consegue manter loop estável com tool-use real quando recebe um problema bem-delimitado, sem custo nem latência explosivos.

**Estado atual** (v0.2): `monitor.ts` faz bucketing e SQL critérios determinístico, depois invoca `detect-divergences` sequencial em representantes, depois `classify-origin` + `suggest-adjustment` em cada divergência, depois `summarize-patterns`.

**Mudança v0.3.1**: depois do bucketing + SQL critérios, `monitor.ts` invoca **um único agente Flue** com:

- **Goal**: "Audite as divergências dos buckets dados. Use as skills disponíveis. Termine com `summarize-patterns` agregando todas as divergências encontradas."
- **Contexto inicial**: gabarito + contexto-momento + lista de buckets + decisões representantes (já samplado pelo monitor).
- **Tools disponíveis**: as 4 skills (`detect-divergences`, `classify-origin`, `suggest-adjustment`, `summarize-patterns`) registradas como tools no Flue, acessíveis via tool-use loop.
- **Tools de I/O**: `read` (limitado a `expected-reasoning/`, `agents-config/`, `references/`) — o agente pode buscar references sob demanda.
- **Limite duro**: `max_tool_calls = 30`, `max_cost_usd = 0.50`. Acima disso, abort + fallback pro pipeline v0.2.

**O que isso testa**:
- Flue mantém o loop sem divergir? (ou modelo entra em recursão tentando re-classificar)
- Tool-use entre skills funciona? (ou Harmony bug ressurge)
- References são acessadas dinamicamente? (validamos finalmente o progressive-disclosure)
- Custo e latência ficam dentro de 2x do pipeline v0.2? (se 5x ou mais, agentic não se justifica nesse ponto)

**Métricas a coletar**:
- Tool calls por run (média, p95)
- Custo por run (em USD)
- Latência total
- Mesmas divergências detectadas vs v0.2? (reproduzir o smoke 5x e comparar conjuntos)
- Robustez: % de runs que terminam dentro do limite

**Critério go/no-go**:
- ✅ **Go pra Stage 2** se: custo ≤ 2x baseline, latência ≤ 2x baseline, qualidade ≥ baseline (mesmas ou melhores divergências detectadas), 0 runs com loop runaway nas últimas 5 execuções.
- ❌ **Abandona ladder** se: custo > 4x, qualidade visivelmente pior, > 1 runaway em 5 runs. Documenta porquê e arquiva o experimento.

### Stage 2 — Sampling de representantes

**Estado atual**: `monitor.ts` aplica `pickRepresentatives(items, k=3, seed)` deterministicamente.

**Mudança v0.3.2**: agente recebe **todas as decisões do bucket** (em vez de 3 reps pré-amostrados) e escolhe quais auditar com base em diversidade ou suspeita.

**Hipótese**: agente faz sampling **mais informativo** (foca em casos limítrofes ou mais divergentes) do que o random-seeded determinístico.

**O que isso testa**: julgamento agêntico sobre dados estruturados sem schema rígido.

**Métricas adicionais**: reps escolhidos cobrem mais variabilidade do bucket? (medido por entropia de campos do lead nos reps escolhidos vs no bucket todo).

### Stage 3 — Tool de query D1

**Estado atual**: `monitor.ts` faz query SQL ao D1 antes de invocar o agente.

**Mudança v0.3.3**: agente recebe descrição do schema D1 (read-only) e **decide quais queries fazer**. Tool `query-decision-log(sql)` exposta com whitelist de operações (SELECT only, max 100 rows, max 5s timeout).

**Hipótese**: agente consegue investigar dados sob demanda quando há sinal ambíguo (ex: "esse padrão é específico desse lead ou aparece em outros? deixa eu ver"). Loops de exploração são úteis.

**Risco**: agente faz queries excessivas, custo de D1 ops explode, queries inválidas. Mitigação: rate limit (max 10 queries/run) + parser SQL pra rejeitar non-SELECT.

### Stage 4 — Agente full

**Estado atual**: `monitor.ts` é entrypoint + checkpoint + carrega gabarito/contexto-momento + bucketing + SQL critérios + escreve artefatos R2 + cria PR + dispara Telegram.

**Mudança v0.3.4**: `monitor.ts` vira ~30 linhas: recebe trigger, dispara o agente com contexto mínimo, persiste o resultado final em R2/PR/Telegram. Tudo no meio é decidido pelo agente:
- Quais decisões puxar do D1
- Como bucketizar (ou se bucketizar)
- Quais critérios SQL aplicar (ou inventar novos)
- Quais skills invocar e em que ordem
- Quando parar e compor PR

**Hipótese**: o "agente final" toma decisões equivalentes ou superiores ao pipeline imperativo, e a perda de controle determinístico é compensada por adaptabilidade (ex: detecta padrões fora dos buckets cartesianos pré-definidos).

**Critério final**: se Stage 4 entregar mesma qualidade que pipeline v0.2 com custo dentro de 5x e latência dentro de 3x, **Flue se justifica**. Se entregar pior ou explodir custo, a tese de "framework agêntico vale a pena" é falsificada (nesse cenário, pelo menos).

## Métricas e instrumentação

Antes de Stage 1, instrumentar baseline v0.2 com:

- **Custo USD por run** (puxado do AI Gateway via API que validamos no v0.2)
- **Latência total + por step**
- **# divergências detectadas + classificadas + sugeridas**
- **Severity escolhida**

Persistir em `monitor-runs/<runId>/metrics.json` automaticamente. Adicionar 1 helper `src/lib/metrics.ts` (~20 linhas).

Cada stage v0.3.X repete a mesma medição. Comparação termina num `docs/V0.3-RESULTS.md` com tabela: `stage | runs | cost_p50 | cost_p95 | latency_p50 | divergences_detected | divergences_matched_baseline | qualitative_notes`.

## Critérios globais de decisão

Ao fim do experimento, decidir entre:

1. **Flue se justifica em Stage N** — adotamos para tudo a partir desse stage. Documenta o porquê.
2. **Flue não se justifica** — Refator pra Workflow CF + fetch direto. Mantém aprendizado documentado em `docs/V0.3-RESULTS.md` como contribuição honesta.
3. **Híbrido** — partes do pipeline ganham com agentic (ex: classify+suggest), partes não (ex: bucketing). Define a fronteira explicitamente.

A decisão deve ser **falsificável** com as métricas coletadas, não baseada em estética ou afinidade com um lado.

## Riscos e mitigações

| Risco | Probabilidade | Mitigação |
|---|---|---|
| Loop runaway (modelo chama tool ad infinitum) | alta | `max_tool_calls` + `max_cost_usd` por run, hard abort |
| Harmony bug ressurge com tool-use real | média | Já trocamos pra Kimi K2.6; se falhar, testar Claude via gateway |
| Custo escala não-linearmente | alta | Stop-loss por run + cap de custo diário no Gateway |
| Divergências do agente diferem do baseline (qualidade pior) | média | Comparação automática vs baseline, se delta > 30% reverte |
| Agente "alucina" classificações sem evidência | média | Manter regras de output schema rígidas + validação valibot pós-chamada |
| Latência > timeout do Worker (10s default, 30s upgrade) | alta | Workflow já necessário pra runs > 30s; pode forçar refator de infra antes mesmo do experimento concluir |

## Out of scope (v0.3 não faz)

- Refator pra CF Workflow (decisão pós-experimento)
- Múltiplos agentes (qualificador + auditor cooperando)
- Fine-tuning de modelo
- Cache de prompts (otimização prematura)
- Custo de produção real (estamos medindo POC com gerador sintético)

## Cronograma sugerido

- Stage 0 (instrumentação baseline): 1 sessão (~2h)
- Stage 1 (handoff post-bucketing): 1-2 sessões
- Stage 2 (sampling agêntico): 1 sessão
- Stage 3 (query D1 agêntica): 1-2 sessões (mais risco de tooling)
- Stage 4 (agente full): 2-3 sessões
- Análise final + V0.3-RESULTS.md: 1 sessão

Total: ~7-10 sessões focadas. Cada stage tem critério go/no-go — pode encerrar mais cedo se a tese cair.

## Princípios não-negociáveis

- **Falsificável** — cada stage tem métrica que decide go/no-go, não "achismo".
- **Reverte rápido** — qualquer stage pode ser desfeito; não faz refator irreversível antes de validar.
- **Compara com baseline** — sem baseline numérico, qualquer mudança vira "parece melhor". Sempre números.
- **Documenta perdas** — se a tese falhar, isso é o aprendizado mais valioso. Não enterrar.
